# BRAINK/KEX Strict Deep Analysis Comment

Use this comment at the top of a request, issue, pull request, or task ticket when the software process must be forced through strict deep analysis, research, refinement, and proof gates.

```text
STRICT_DEEP_ANALYSIS_REQUIRED:
Anchor this task to a.keddeh / BRAINK / KEX. Treat every important word in the request as a function, not decoration. Before implementation, perform deep context inspection of existing files, prior artifacts, constraints, and available evidence. Do not generate a solution from assumption when repository evidence, executable checks, source-backed research, or mathematical/technical derivation is required.

Required process:
1. LANGUAGE -> MEANING -> FUNCTION -> CONSTRAINT -> ACTION -> PROOF -> STATUS.
2. Define the exact objective, non-objectives, environment, state, function, whole-system boundary, and pending gates.
3. Inspect local repository artifacts first; use external research only when facts are current, high-stakes, source-dependent, or not locally provable.
4. Convert the request into a plan with targets, commands/actions, expected outputs, verification methods, failure routes, and rollback/repair route.
5. Refine the software process to the highest ability permitted by available software, technology, local environment, credentials, time, and a.keddeh calibration.
6. No promoted claim may be marked complete without evidence: artifact, executable result, derivation, measurement, or source-backed validation.
7. Separate MODEL-LOCAL proof from EXTERNALLY-UNVALIDATED claims. Do not call simulation hardware proof, local proof external adoption, theory physical law, or a plan an implementation.
8. If evidence is missing, mark the exact gate PENDING instead of guessing.
9. If safety, authorization, privacy, medical, legal, financial, or identity risk appears, route into defensive analysis, consent, and boundary preservation.
10. Final output must record what was required, what was done, what proves it, what failed, and what remains pending.

Acceptance gate:
The work is not acceptable until the repository diff, tests/checks, evidence, constraints, and pending gates are all explicitly reported.
```

## Recursive failure rule

When a problem recurs across sessions without resolution, do not attempt the same approach again. Instead:

1. Identify the exact step that failed and why.
2. Change the approach before retrying.
3. If the root cause is a missing prerequisite (e.g. a file, access, or artifact), establish the prerequisite first.
4. Record the failure, the new approach, and the gate that proves it resolved.
